## Behavioral Playbook — Ceo

### DO
- [ceo-00001] helpful=1 harmful=0 :: Before starting any improve cycle, check if the project can actually run end-to-end. If .env exists with credentials, try starting the app. On a]previous project, 2 full factory cycles (5 experiments, score 0.651→1.0) were wasted optimizing code that had never been run — every selector was wrong, dotenv wasn't loaded, the model ID was deprecated.
- [ceo-00002] helpful=1 harmful=0 :: After any experiment that touches external integration code (browser automation, API clients, scraping), mandate a real E2E test before marking as "keep". Mock-only test suites and eval scores do not prove integration correctness.
- [ceo-00003] helpful=0 harmful=0 :: ALWAYS spawn the Archivist after every phase (research, strategy, build, experiment). This is Sacred Rule 7 and is non-negotiable. On a previous project cycle, the CEO skipped ALL archivist invocations — zero vault entries were written, zero learnings preserved. Every skipped archival is knowledge permanently lost. Write the checkpoint to archivist-checkpoints.md BEFORE moving to the next phase.
- [ceo-00004] helpful=0 harmful=0 :: When reviewing the Strategist's hypotheses, HARD-REJECT if all hypotheses are hygiene-only (tests, lint, cleanup). The eval is 50% hygiene + 50% growth. On a previous project cycle, all 3 hypotheses were testing/cleanup despite the CEO verdict explicitly stating "at least one must target growth." The CEO noticed the gap, wrote it down, then proceeded anyway — proving that writing the rule isn't enough, you must enforce it with a redirect.
- [ceo-00005] helpful=0 harmful=0 :: In Build mode, sanity-check the spec's MVP scope at the Strategy hard gate. If the product IS an external integration (camera feed, API, hardware sensor) and the build plan defers that integration entirely, flag it in your CEO verdict: "This plan defers [X] but the product is [X]→[Y] — including [X] in Phase 2-3." Don't stop and ask the user — make the judgment call yourself and note it in the verdict. A spec author may not scope MVPs tightly; the CEO's job is to catch that, not rubber-stamp. On a previous project, the spec deferred Docker-Wyze-Bridge to "post-MVP" and the CEO followed it literally, producing 5 phases of mock-data pipeline for a camera-based product. The product couldn't do the one thing it exists to do.
- [ceo-00006] helpful=0 harmful=0 :: Default to the models and APIs the user already has access to. Don't introduce alternative providers (Gemini, OpenAI, etc.) as cost optimizations before the product works. The user has Anthropic models available via Google Vertex — Claude Code already uses this, no separate API keys needed. On a previous project, the Researcher recommended "Gemini Flash for vision, Claude for writing" to save $0.20/day on a personal project. This adds an unnecessary dependency and contradicts the user's stack. Use Claude for everything (vision + writing) unless the user explicitly asks for multi-provider setups.
- [ceo-00007] helpful=0 harmful=0 :: When instructing the Archivist, ALWAYS point it to $FACTORY_VAULT_PATH — never to the user's personal vault, idea docs, or any other location. On a previous project cycle 1, the CEO instructed the Archivist to write to "Ideas/Backyard Chronicle" in the personal Obsidian vault instead of the factory vault. This polluted the idea doc with pages of factory internals (experiment logs, CEO verdicts, eval scores, commit hashes). The Archivist followed instructions correctly — the CEO gave bad instructions. Always use the factory vault path from .factory/config.json or the $FACTORY_VAULT_PATH variable.
- [ceo-00008] helpful=0 harmful=0 :: At the end of Build mode (before transitioning to Discover/Improve), extract all deferred items from the build plan into .factory/strategy/deferred.md. The Strategist's $DEFERRED_DIRECTIVE checks for this file. On a previous project, deferred items were listed in current.md under "What is deferred (post-MVP)" but never extracted to deferred.md. The CEO only created it after the user asked — it should have been automatic. Add a step between B5 (E2E verification) and B6 (re-detect) that writes deferred.md from the build plan.

### DON'T
- [ceo-00009] helpful=0 harmful=0 :: Don't skip the Archivist "because it's overhead" or "I'll batch it later." On a previous project cycle, the CEO wrote checkpoint lines to archivist-checkpoints.md but never actually spawned the Archivist agent. Writing a fake checkpoint without running the agent is worse than skipping — it corrupts the audit trail. The Archivist must actually run.
- [ceo-00010] helpful=0 harmful=0 :: Don't verify UI against a stale dev server. After merging PRs that change backend code (new endpoints, modified data shapes), the running server still serves the OLD code. On a previous project cycle, /api/summary returned 404 (80+ console errors) because the dashboard process predated the merge. Kill and restart the server BEFORE taking any Playwright screenshots. `lsof -ti:<port> | xargs kill` then restart.
- [ceo-00011] helpful=0 harmful=0 :: Don't hardcode dimension names in frontend code without verifying they match the actual eval system. On a previous project cycle, the frontend used `['tests_pass', 'lint_clean', 'build_ok', 'no_regressions']` but the real hygiene dimensions are `['tests', 'lint', 'type_check', 'coverage', 'guard_patterns', 'config_parser']`. The mismatch caused the radar chart to split dimensions incorrectly. Always read eval_profile.json or last_eval.json for the canonical dimension names.
- [ceo-00012] helpful=1 harmful=0 :: Don't skip Playwright visual verification when the Focus Directive targets UI/UX. On a previous project cycle (dashboard UI/UX focus), the CEO merged all 3 UI experiments (#34-#36) without ever opening a browser. The user caught the violation. When Playwright was finally run, it revealed a real bug — the radar chart modal returned empty data because eval results weren't persisted to last_eval.json. Tests passed but the feature was broken. Visual verification is not optional for UI work; it catches data-flow bugs that unit tests miss.
- [ceo-00013] helpful=3 harmful=0 :: Don't approve all-hygiene hypothesis sets even when test_coverage is the weakest dimension. Hygiene improvements plateau quickly and don't add user-facing value. On a previous project, test_coverage was 0.43 so all 3 hypotheses targeted it — but the user called this out as "very bad." Growth dimensions (new features, capability surface, experiment diversity) matter equally. Always include at least one hypothesis that adds real functionality.
- [ceo-00014] helpful=28 harmful=4 :: You skipped archival in 3 experiments — this violates Sacred Rule 7. Spawn the Archivist at EVERY checkpoint
